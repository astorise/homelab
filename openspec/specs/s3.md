# S3 Storage Service (home-s3)

## Purpose
Embedded S3-compatible object storage using RustFS. Runs as a Windows service,
accessible from Windows localhost and from WSL pods via a stable routing chain.
Managed via gRPC over Named Pipe.

---

### Requirement: Loopback-only binding
The service MUST bind exclusively on `127.0.0.1:9000`.
It MUST NOT bind on `0.0.0.0` or the Hyper-V interface directly,
to prevent exposure on the LAN or WAN.

#### Scenario: RustFS startup with loopback endpoint
- GIVEN config `endpoint: http://127.0.0.1:9000`
- WHEN the service starts
- THEN RustFS listens only on `127.0.0.1:9000`
- AND the port is not reachable from the physical network

#### Scenario: Non-loopback endpoint causes hang
- GIVEN config `endpoint: http://172.18.192.1:9000` (Hyper-V IP)
- WHEN the service starts
- THEN RustFS fails to initialize or hangs indefinitely
- BECAUSE RustFS does not support binding on non-loopback addresses

---

### Requirement: WSL pod access via stable loopback alias
WSL pods MUST be able to reach home-s3 via `s3.wsl:9000`, resolving to
the stable WSL loopback alias `10.255.255.254` without depending on
the dynamic Hyper-V IP.

#### Scenario: Pod connects to S3
- GIVEN DNS `s3.wsl A 10.255.255.254` is configured
- AND iptables PREROUTING DNAT `10.255.255.254:9000 → <gw>:9000` is active in WSL
- AND Windows portproxy `<gw>:9000 → 127.0.0.1:9000` is active
- WHEN a pod connects to `http://s3.wsl:9000`
- THEN the connection reaches home-s3 on Windows loopback
- AND an HTTP response (e.g. 403 Forbidden without credentials) is returned

---

### Requirement: Hyper-V portproxy lifecycle management
The homelab app MUST create and maintain a Windows portproxy rule
`<hyper-v-ip>:9000 → 127.0.0.1:9000` on the `vEthernet (WSL)` interface.
This rule MUST be updated when the Hyper-V IP changes.

#### Scenario: Hyper-V IP changes
- GIVEN previous portproxy was for `172.18.192.1:9000`
- WHEN `sync_home_s3_wsl_binding()` detects new Hyper-V IP `172.18.195.1`
- THEN the old portproxy rule is deleted
- AND a new portproxy `172.18.195.1:9000 → 127.0.0.1:9000` is created

---

### Requirement: WSL iptables DNAT at boot
The WSL k3s-init.sh MUST install iptables DNAT rules at boot so that
connections to `10.255.255.254:9000` from pods are forwarded to the
default gateway (Hyper-V Windows IP).

#### Scenario: WSL instance reboots
- GIVEN `/etc/k3s-env` contains `WSL_S3_LOOPBACK_IP=10.255.255.254` and `WSL_S3_PORT=9000`
- WHEN k3s-init.sh runs `ensure_s3_iptables_dnat()`
- THEN `iptables -t nat -A PREROUTING -d 10.255.255.254 -p tcp --dport 9000 -j DNAT --to-destination <gw>:9000` is set
- AND `iptables -t nat -A OUTPUT ...` is set (for WSL-host-originating traffic)
- AND `iptables -t nat -A POSTROUTING -d <gw> -p tcp --dport 9000 -j MASQUERADE` ensures return packets reach the pod

#### Scenario: iptables binary discovery
- GIVEN busybox `/sbin/iptables` does not have the iptables applet
- WHEN `find_iptables()` is called
- THEN the real binary at `/var/lib/rancher/k3s/data/<hash>/bin/aux/iptables` is used

---

### Requirement: Config file at well-known path
The service MUST read its configuration from `C:\ProgramData\home-s3\s3-config.json`
at startup. Changing the endpoint requires a service restart (RustFS uses startup config).

#### Scenario: Config reload warning
- GIVEN the config file is updated while the service is running
- WHEN `reload_config` RPC is called
- THEN the service logs a warning that RustFS keeps the startup configuration
- AND the new config takes effect only after a full service restart
